# Plan: runtime-image

**Date:** 2026-04-18T14-26-35 | **Based on:** 2026-04-18T14-18-25_research.md | **Branch:** feat/runtime-image

## Architecture

### Summary
Ship a single multi-stage Docker image that lets any developer point viva at their own codebase with one `docker run` command. Stage 1 (`node:20-bookworm-slim`) builds the viewer with sourcemaps off; stage 2 (`python:3.12-slim-bookworm`) installs the crawler, copies the pre-built viewer assets, and runs a shell entrypoint that crawls the read-only `/target` mount then serves `/app/viewer/dist/` via `python -m http.server`. Zero source code changes except a one-line sourcemap env-var gate in `viewer/vite.config.ts`. Zero network at runtime. Re-crawl on every run — no cache volume, no state, simplest possible UX.

### Data Flow

```
[user's codebase on host]
  |
  | docker run --rm -v "$(pwd):/target:ro" -p 5173:5173 ghcr.io/sgupta604/viva [crawler args...]
  v
[container: python:3.12-slim-bookworm]                (multi-stage build: Node only in BUILD stage, absent at RUNTIME)
  |
  | /app/docker/entrypoint.sh  (bash, set -euo pipefail)
  |   1. python -m crawler /target --out /app/viewer/dist/graph.json --no-timestamp -v "$@"
  |   2. echo "Visualizing /target -> http://localhost:5173"
  |   3. exec python -m http.server 5173 --bind 0.0.0.0 --directory /app/viewer/dist
  v
[host port 5173]  <-- bind mount :ro (read-only); re-crawl every run; offline RUNTIME guarantee intact
  |
  v
[browser]  loads / + /graph.json, explores visualizer
```

**Explicit callouts:**
- **Multi-stage build:** `node:20-bookworm-slim` only in Stage 1 to run `npm ci && VITE_SOURCEMAP=0 npm run build`. Final image has **no Node, no npm, no node_modules**. Viewer ships as pre-built static `dist/`.
- **Read-only mount:** `:ro` documented prominently. Crawler never writes to `/target` by design (output goes to `/app/viewer/dist/graph.json`, inside the container layer).
- **Re-crawl every run:** No cache volume, no stale-graph confusion, no `--rebuild` flag. Matches "really really really simple".
- **Offline-RUNTIME guarantee intact:** No curl/wget/fetch in entrypoint. No telemetry. BUILD stage fetches npm/pip registries (normal); RUNTIME fetches nothing. Same non-negotiable as v1 viewer.

## Key Decisions

- **Multi-stage Dockerfile over single-stage.** Keeps Node (~200 MB + node_modules) out of the runtime layer. Final image well under 300 MB target.
- **`python:3.12-slim-bookworm` base.** lxml 5.x ships manylinux amd64 wheels; no `build-essential` needed. Bookworm codename pinned (no `:latest`, no floating `:3.12`).
- **`python -m http.server` over `nginx:alpine`.** Zero new dependency (Python already present for crawler). Viewer is a pure static SPA — no SPA fallback routing. Simplest possible server.
- **Bash entrypoint, not Python.** Shell is natural for "run A, print B, exec C" sequencing. `exec` ensures SIGTERM propagation.
- **Re-crawl every run, no cache volume.** Matches "really really really simple" — no stale-graph confusion, no `--rebuild` flag, no named volume to manage.
- **`"$@"` forwards to crawler, not server.** Power-user knob is include/exclude/emit-sources — server flags are irrelevant.
- **Sourcemap gate via env var in `vite.config.ts`.** One-line diff, default stays ON (dev workflow untouched), container build passes `VITE_SOURCEMAP=0`. Strips ~43 MB from image.
- **amd64-only MVP, arm64 follow-up.** Cuts buildx/qemu setup out of this feature. Mac M1/M2 users get platform-mismatch warning + emulation; documented.
- **End-user Docker docs ABOVE contributor devcontainer docs in root README.** End users are the primary audience.

## Notes for Implementer

- **Windows line endings.** Commit `docker/entrypoint.sh` with LF only. `.gitattributes` rule (T1) enforces this on future clones. If you're on Windows, verify with `file docker/entrypoint.sh` or `cat -A docker/entrypoint.sh | head` before committing.
- **Exec bit on Windows.** `git update-index --chmod=+x docker/entrypoint.sh` after `git add` — Windows filesystems don't track the exec bit, but git does. Without this step, the Dockerfile's `ENTRYPOINT` will fail with "permission denied" inside the container.
- **Non-editable pip install.** Use `pip install --no-cache-dir ./crawler`, NOT `pip install -e ./crawler` — editable installs add a `.egg-link` layer and don't gain us anything in an immutable image.
- **No `pip install .[dev]`.** Only runtime deps (`lxml`, `ruamel.yaml`). Dev extras (pytest, ruff) bloat the image.
- **Don't `COPY . .`.** Copy `crawler/` and the viewer `dist/` from the build stage explicitly. Keeps layers clean and cacheable.
- **Don't `mkdir /target` in the Dockerfile.** The bind mount creates it. Baking it in invites "is /target expected to be populated at build time?" confusion.
- **Don't add `CMD`.** `ENTRYPOINT ["/app/docker/entrypoint.sh"]` + `"$@"` forwarding is enough. A separate `CMD` just muddies the arg-passthrough story.
- **Sourcemap env var name.** Use `VITE_SOURCEMAP` (research Q5). Value `"0"` = off; anything else or unset = on. This keeps the boolean test simple and the default ON.
- **Smoke test fixture.** `crawler/tests/fixtures/sample-module` is the committed fixture from v1. Reuse it for T7's container smoke test — deterministic, small, exercises multiple parsers.
- **If `docker build` fails on your machine for arm64 wheel reasons:** you're on Mac M1/M2 without platform flag. Add `--platform linux/amd64` to the build and run commands. Document this gotcha in T5 troubleshooting if it surfaces.
- **HH-MM-SS vs HH:MM:SS filename.** This plan uses `HH-MM-SS` (Windows-safe). Research flagged WORKFLOW.md still uses `HH:MM:SS` in its examples; that's a separate pipeline-infra fix, not this feature's concern.
