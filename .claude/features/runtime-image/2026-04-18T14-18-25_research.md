# Research: runtime-image

**Date:** 2026-04-18T14-18-25 | **Status:** research-complete

## Goal

Ship a single Docker image that lets any developer point viva at their own codebase with one command — `docker run --rm -v "$(pwd):/target:ro" -p 5173:5173 <image>` — and get an interactive visualizer in their browser. Zero host installs, zero prior viva knowledge, zero configuration. Image is the distribution artifact for end users; the devcontainer story (previously researched then abandoned) is explicitly out of scope.

## Requirements

### Functional
- [ ] FR1: `docker run --rm -v "$(pwd):/target:ro" -p 5173:5173 <image>` is sufficient — no additional flags, env vars, pre-crawl, or volume config required. (user brief "baseline UX target")
- [ ] FR2: On start, entrypoint (a) crawls `/target` into a graph.json co-located with the viewer assets, (b) starts a static HTTP server on `0.0.0.0:5173`, (c) prints `Visualizing /target -> http://localhost:5173` in clear, human-friendly form before blocking. (user brief Q1)
- [ ] FR3: Additional args after the image name are forwarded to `python -m crawler` so power users can pass `--include`, `--exclude`, `--emit-sources`, `-v`, etc. Default invocation with no extra args uses sensible defaults only. (user brief Q7)
- [ ] FR4: Re-crawl happens on every `docker run` — no re-crawl-skipping cache, no named volume required, no state between runs. Simplest default. (user brief Q1: "Lean toward simpler default unless re-crawl is noticeably slow")
- [ ] FR5: Container serves the already-built viewer `dist/` unchanged; no dev server, no Vite, no Node at runtime. (user brief Q3)
- [ ] FR6: Mount is documented as read-only (`:ro`) in the user-facing command. Crawler never writes to `/target` — verified by existing design (emit target defaults to `/app/viewer/dist/graph.json`, inside the image layer). (user brief Q5)
- [ ] FR7: Crawler's existing non-fatal handling of per-file issues is preserved — `PermissionError` in `discovery.py:63-65` is already swallowed; `XMLSyntaxError` etc. are captured into `FileNode.parse_error` per `crawler/src/crawler/parsers/xml.py:28-33`. No change needed; behavior matches "warn, don't crash". (user brief Q5)
- [ ] FR8: End-of-crawl summary log is visible to the user: current `__main__.py:63-70` already logs "crawled <root>: N files, M edges (K unresolved, P parse errors)" at INFO. The entrypoint must set log level to INFO (default is WARNING) so this line shows up without `-v`. (user brief Q9)
- [ ] FR9: If the crawl produces a graph substantially larger than v1 scope (>~500 files — rough 10x multiplier over v1's 20-50), print a one-line warning ("v1 is tuned for ~20-50 files; viewer may lag at this size") but continue. Never block. (user brief Q6)
- [ ] FR10: Image carries zero network capability at runtime — no curl, no fetch, no telemetry on container start. Same offline guarantee as v1 viewer. (spec "non-negotiable"; CLAUDE.md "Offline guarantee")
- [ ] FR11: Root `README.md` gets a short "Run with Docker" section (5-10 lines) showing the one-liner and pointing at `docker/README.md` for detail. Platform-specific notes (Windows PowerShell quoting, macOS path mount) kept short.
- [ ] FR12: `docker/README.md` (or similar path, plan-agent decides) documents: the one-liner, the optional crawler args, `--help` passthrough, the no-Docker fallback (point at v1 install instructions), and the "proprietary code never leaves the container — mount is `:ro`, no network" assurance. (user brief Q11)

### Technical
- [ ] TR1: Multi-stage Dockerfile. Stage 1 `node:20-bookworm-slim` builds the viewer (`npm ci && npm run build` -> `viewer/dist/`). Stage 2 `python:3.12-slim-bookworm` installs the crawler (`pip install -e ./crawler`) and COPYs `viewer/dist/` from stage 1. No Node in the final image. (user brief Q3)
- [ ] TR2: Final image base `python:3.12-slim-bookworm`. Slim is acceptable here (not the devcontainer trap) because lxml 5.x ships manylinux wheels for cpython 3.12 on amd64 and arm64 — no compiler needed, no build-essential layer. If wheel resolution ever fails, add `build-essential libxml2-dev libxslt1-dev` conditionally; verify during T-validation before accepting slim. (devcontainer research risk table, reinterpreted for runtime)
- [ ] TR3: Static file server uses `python -m http.server 5173 --bind 0.0.0.0 --directory /app/viewer/dist`. Zero extra dependency (Python is already in the image). The viewer is a pure static SPA with no path-based routing, so `http.server`'s lack of SPA fallback is fine. (user brief Q2)
- [ ] TR4: Entrypoint is a small shell script (`/app/entrypoint.sh`), `#!/usr/bin/env bash`, `set -euo pipefail`. Orchestrates: crawl -> print URL -> exec http.server. Receives any passed args and forwards to crawler. `exec` on the final server call so signals propagate (Ctrl+C stops cleanly). `.gitattributes` must already force LF+exec on `.sh`; extend if not present. (user brief, CLAUDE.md)
- [ ] TR5: `HEALTHCHECK` optional — `curl -f http://localhost:5173/graph.json || exit 1` — but curl is not in slim by default; either install it (+500 KB) or use `python -c "import urllib.request; urllib.request.urlopen('http://localhost:5173/graph.json')"`. Lean toward healthcheck-omitted for MVP to keep image minimal. Defer. (non-critical)
- [ ] TR6: `.dockerignore` at repo root excludes `viewer/node_modules/`, `viewer/dist/`, `crawler/.venv/`, `crawler/.pytest_cache/`, `.claude/`, `.git/`, `docs/`, `*.md`, `.vscode/`, `.idea/`, tests (`crawler/tests/`, `viewer/e2e/`), `viewer/test-results/`, `viewer/playwright-report/`, `**/__pycache__`, `viewer/*.tsbuildinfo`. Keeps build context small and prevents leaking host artifacts. (standard Docker hygiene)
- [ ] TR7: Image labels for provenance — `org.opencontainers.image.title=viva`, `.source=<repo>`, `.version=0.1.0`, `.description=Config codebase visualizer (offline)`. (standard OCI)
- [ ] TR8: Target image size <300 MB uncompressed. Budget: python-slim base (~45 MB) + pip deps (lxml ~10 MB + ruamel.yaml ~1 MB) + crawler source (<1 MB) + viewer dist without sourcemaps (~15 MB, see finding below) = well under budget. (user brief risks)
- [ ] TR9: `vite build --sourcemap false` (or equivalent flag override) for the image build stage only — source maps (~43 MB of dist/) are a local dev affordance and waste bytes in the runtime image. v1 `vite.config.ts:20-21` has `sourcemap: true` hard-coded; plan-agent must decide whether to pass `--mode production` with an override or to set an env-driven conditional in vite.config.ts. Either way, DO NOT delete the existing v1 setting — dev workflow relies on it.
- [ ] TR10: Multi-arch build is a plan-time decision. Document the `docker buildx build --platform linux/amd64,linux/arm64 -t ... --push .` command. MVP MAY ship single-arch amd64 and note arm64 as follow-up — call out the Mac M1/M2 warning-on-pull UX cost so plan-agent chooses deliberately. (user brief Q4)

### Constraints
- No changes to `graph.json` schema. No changes to `crawler/src/` source (entrypoint orchestrates the existing CLI, nothing more). Viewer `src/` untouched.
- No network fetch at container runtime — any `curl`/`wget` in the entrypoint is forbidden.
- No baked credentials, no registry secrets in the image.
- Windows filename convention: pipeline artifacts use `HH-MM-SS` dashes. (v1-demo SUMMARY "Deferred Items" — still unresolved in WORKFLOW.md)

## Code Examples from Spec

**Target user flow (one command, first time and every time):**
```bash
docker run --rm -v "$(pwd):/target:ro" -p 5173:5173 ghcr.io/sgupta604/viva
# prints: Visualizing /target -> http://localhost:5173
# user opens browser manually
```

**Power-user passthrough (forward args to crawler):**
```bash
docker run --rm -v "$(pwd):/target:ro" -p 5173:5173 ghcr.io/sgupta604/viva \
  --exclude '**/tests/**' --emit-sources
```

**Entrypoint skeleton (target shape, not final):**
```bash
#!/usr/bin/env bash
set -euo pipefail
python -m crawler /target \
  --out /app/viewer/dist/graph.json \
  --no-timestamp \
  -v \
  "$@"
echo "Visualizing /target -> http://localhost:5173"
exec python -m http.server 5173 --bind 0.0.0.0 --directory /app/viewer/dist
```

**Multi-stage Dockerfile skeleton:**
```dockerfile
# Stage 1 — build viewer
FROM node:20-bookworm-slim AS viewer-build
WORKDIR /src/viewer
COPY viewer/package*.json ./
RUN npm ci
COPY viewer/ ./
RUN npm run build  # or a sourcemap-free variant

# Stage 2 — runtime
FROM python:3.12-slim-bookworm
WORKDIR /app
COPY crawler/ ./crawler/
RUN pip install --no-cache-dir -e ./crawler
COPY --from=viewer-build /src/viewer/dist ./viewer/dist
COPY docker/entrypoint.sh /app/entrypoint.sh
RUN chmod +x /app/entrypoint.sh
EXPOSE 5173
ENTRYPOINT ["/app/entrypoint.sh"]
```

**Smoke test (end-to-end, post-build):**
```bash
docker build -t viva:test .
docker run -d --rm --name viva-smoke \
  -v "$PWD/crawler/tests/fixtures/sample-module:/target:ro" \
  -p 5173:5173 viva:test
# wait for startup (~3s)
curl -fsS http://localhost:5173/graph.json | python -c "import json,sys; g=json.load(sys.stdin); assert g['version']==1; assert len(g['files'])>0"
curl -fsS http://localhost:5173/ | grep -q '<div id="root"'
docker stop viva-smoke
```

## Patterns to Follow

- **Pinned image tags** (devcontainer research "Patterns to Avoid": `latest` → bitrot). Use `python:3.12-slim-bookworm` and `node:20-bookworm-slim` — both pinned by distro codename, not floating `:latest` or `:3.12`.
- **BUILD vs RUNTIME offline distinction** (load-bearing in v1 and re-usable here). The image BUILD step may fetch from npm, pip, and apt registries — normal. The image RUNTIME must not fetch anything. Put this in `docker/README.md` explicitly. Same principle used in devcontainer-abandoned's README.
- **One concern per file** (CLAUDE.md "One component per file"). Don't cram entrypoint logic into `CMD ["bash","-c","..."]` inside the Dockerfile — use a separate `entrypoint.sh`.
- **Deterministic crawler runs** (v1 SUMMARY "Committed fixture graph.json"). Entrypoint invokes crawler with `--no-timestamp` so repeated runs against the same target are byte-identical — useful for smoke-test diffing and matches the fixture's expected output format.
- **Orchestrator dispatches, doesn't code** (CLAUDE.md non-negotiable). This feature creates Dockerfile + shell + README. No `crawler/` or `viewer/` source edits, except TR9 if vite config needs an env-driven sourcemap toggle — flag this clearly for plan-agent so the right specialist gets the task (frontend-agent for vite.config.ts, execute/plan-level for the rest).
- **LF line endings + exec bit on `.sh` files** (devcontainer-abandoned T1). Must survive a Windows clone. Same `git update-index --chmod=+x docker/entrypoint.sh` after first commit.
- **`exec` the final long-running process in the entrypoint** so SIGTERM from `docker stop` propagates. Standard container idiom.

## Patterns to Avoid

- **`--no-cache` as a default `pip install` flag** in the runtime stage — common anti-pattern; prefer `--no-cache-dir` to avoid bloating the image layer. (Don't confuse with `--no-cache`.)
- **Running the viewer via `vite preview` or `npm run dev` in the container** (user brief Q2). No Node at runtime; no dev server. Static serve only.
- **`pip install -e .[dev]`** in the runtime stage. Dev deps (pytest, ruff) are not needed to run the crawler. Use plain `pip install -e ./crawler` or better `pip install ./crawler` (non-editable for immutable layers).
- **Baking `/target` into the image.** The mount point is established at `docker run` time via `-v`. Image must not assume `/target` exists at build time.
- **Creating `/target` in the Dockerfile via `RUN mkdir /target`.** Unnecessary — the bind mount creates it. Adding it invites confusion about "is it expected to be populated at build time".
- **Telemetry / phone-home on container start** (user brief non-goals). No `curl`/`wget` in entrypoint. Offline guarantee is absolute at runtime.
- **Shipping sourcemaps in the image** (they're 43 MB of 58 MB total dist). End users have no browser devtools attached to this; they're dead weight.
- **Hardcoding a single platform.** Even if multi-arch is deferred to follow-up, the build invocation docs should use `docker buildx` rather than `docker build` so cross-platform builds are a one-flag change, not a toolchain swap.
- **Adding CI publish workflow in this feature** (user brief Q10 + non-goals). Document `docker push` as the manual step; a GitHub Actions workflow is a clean follow-up feature.
- **Re-crawl-skipping cache using named volumes** (user brief Q1). Adds complexity the user must reason about ("why is my graph stale?"). The simplest UX wins — re-crawl on every run. Crawler is fast enough at v1 scope.

| Question | Resolution |
|----------|------------|
| Q1: Where should the entrypoint file live — `docker/entrypoint.sh` or `scripts/entrypoint.sh`? | **RECOMMEND `docker/entrypoint.sh`.** `docker/` co-locates everything Docker-specific (entrypoint, README, maybe a future `compose.yaml`). `scripts/` in devcontainer-abandoned was for dev-host tooling; this is container-internal. Clear separation of concerns. |
| Q2: Should `graph.json` be written into `/app/viewer/dist/graph.json` (inside image layer) or to a `/data/graph.json` that's served from a separate path? | **RECOMMEND write into `/app/viewer/dist/graph.json`.** The viewer loads `/graph.json` via fetch (v1 design — see `viewer/src/lib/graph/load.ts`). Putting it next to `index.html` means no server config, no URL rewriting, no env-var plumbing. Writing to a container-layer path (not a mounted volume) is fine because `--rm` cleans up, and crawler is fast enough to re-run. |
| Q3: Should the entrypoint print the URL before or after the crawl completes? | **RECOMMEND AFTER.** If the URL prints first, user opens the browser, sees a blank viewer, fetch of `/graph.json` 404s because crawl is still running, user gets confused. Better: crawl, then print "Visualizing /target -> http://localhost:5173", then start the server. Crawl on a 50-file module takes <2s; user waits for the line, knows it's ready. Print a "Crawling /target..." line at the start so the user sees something. |
| Q4: Multi-arch in MVP or follow-up? | **RECOMMEND MVP = amd64 only, arm64 as follow-up.** Cuts CI/buildx setup out of this feature. Mac M1/M2 users will see a "platform mismatch" warning on pull but the image runs via emulation. Document this; if Mac adoption is high, bump arm64 to its own small feature. Plan-agent can override if they have buildx set up easily. |
| Q5: Should we override `vite build`'s `sourcemap: true` (vite.config.ts:20-21) for the image? | **RECOMMEND YES, via env var.** Add `sourcemap: process.env.VITE_SOURCEMAP !== 'false'` to vite.config.ts; the image build stage passes `VITE_SOURCEMAP=false`. Keeps dev workflow untouched (default true), strips 43 MB from the image. Small, clean change — 1 line in vite.config.ts. |
| Q6: Healthcheck? | **RECOMMEND DEFER.** Adds curl or a python one-liner as HEALTHCHECK CMD. Small complexity; no clear user need for MVP (user sees the URL print, opens browser, gets immediate feedback). Good candidate for a post-MVP quickfix. |
| Q7: Which registry for distribution? | **DEFER — documentation only.** Plan-agent documents `docker build -t viva .` locally and notes that pushing to `ghcr.io/sgupta604/viva` (preferred, free, tied to GitHub auth) or Docker Hub is the user's step. A GitHub Actions publish workflow is a clean follow-up feature, not part of this one. |
| Q8: Should we provide a `docker-compose.yml` for the one-liner? | **RECOMMEND NO for MVP.** `docker run` is already one line. Compose adds a file to read, a `docker compose up` command to remember, and an illusion of complexity. Keep the brief's "really really really simple" goal. |
| Q9: Should we print color / spinner during the crawl? | **RECOMMEND NO for MVP.** Python logging plain text is fine. ANSI color works in most modern terminals but doesn't on Windows cmd.exe without setup; adds zero signal. |
| Q10: How to document the "mount read-only" in the printed output? | **RECOMMEND print nothing about `:ro`.** The one-liner the user runs includes it; if they omit it, the tool still works. Don't nag. The docs mention it; the runtime doesn't. |

## Recommended Approach

Single stream, target 8-12 tasks. Deliver a multi-stage Dockerfile + entrypoint shell + docker-README + root-README pointer, with one small edit to `viewer/vite.config.ts` for the sourcemap toggle.

**Tasks roughly** (plan-agent refines):
1. `.gitattributes` rule for `*.sh text eol=lf` + `.dockerignore` file at repo root.
2. `viewer/vite.config.ts` sourcemap env-var toggle (frontend-agent).
3. `Dockerfile` — multi-stage, viewer build → python runtime.
4. `docker/entrypoint.sh` — crawl, print URL, exec http.server.
5. `docker/README.md` — user-facing one-liner + args + platform notes + offline proof.
6. Root `README.md` — short "Run with Docker" section, link to `docker/README.md`.
7. Smoke test script (bash; optional but recommended) — build + run against sample-module fixture + curl-verify graph.json + HTML loads + stop.
8. Image size + label check.
9. Validation (shellcheck on entrypoint, dockerfile lint, build succeeds, smoke test green).

**Why this over alternatives:**
- **Multi-stage over single-stage:** Keeps Node out of the runtime layer — cuts ~200 MB and aligns with the "slim is possible because the viewer is pre-built" insight.
- **`python -m http.server` over `nginx:alpine`:** Zero new dependency, no nginx config file, no second process to manage. Viewer is pure static SPA — no SPA fallback routing needed. `http.server` is MVP-perfect.
- **Re-crawl every run, no cache:** Matches "really really really simple" — no volume to manage, no stale-graph confusion, no `--rebuild` flag. Crawl is fast.
- **Bash entrypoint over Python entrypoint:** Shell is natural for "run A, print B, exec C" sequencing. No need for a second Python process just to orchestrate two Python commands.
- **Single-arch amd64 first, arm64 follow-up:** Time-boxed; covers Windows+Linux and most Mac users via emulation. Keeps this feature small.
- **Entrypoint forwards `"$@"` to crawler, not to the server:** The common power-user knob is crawler flags (include/exclude/emit-sources). Server flags are irrelevant. Gives power users the right knob with zero UX tax on the default case.
