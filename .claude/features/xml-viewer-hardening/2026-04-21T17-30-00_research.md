# Research: xml-viewer-hardening

**Date:** 2026-04-21T17:30:00 | **Status:** research-complete

> Filename note: Windows NTFS forbids `:` in filenames, so time portion uses `-`. Lex sort preserved.

## Goal

Harden viva against real-world proprietary config codebases uncovered by dogfooding: fix the XInclude (`xi:include`) parse failure, ship file sources by default so "source not shipped" never surfaces unprompted, add view-mode/sort controls to the viewer so users aren't stuck with a single grouped graph, and tighten crawler perf + ref coverage. Test via an actual crawl of the viva repo itself.

## Feature Classification

Cross-cutting (crawler + viewer) enhancement on an existing v1. Not greenfield — builds on `crawler/src/crawler/parsers/xml.py`, the Zustand viewer stores, and the current `FilterBar`/`GraphCanvas`/`FileDetailPanel` triad. One branch, one PR (user brief).

## Code Examples / Concrete Findings

### 1. Root-cause: lxml xi:include failure

Current code at `crawler/src/crawler/parsers/xml.py:27`:
```python
tree = etree.fromstring(data)          # strict parser, no recover
```

Reproduction of the bug + fix, verified in this research:
```python
xml = b'<?xml version="1.0"?><root><xi:include href="foo.xml"/></root>'
# Strict (current):
etree.fromstring(xml)
# → XMLSyntaxError: Namespace prefix xi on include is not defined
# Recover:
parser = etree.XMLParser(recover=True)
tree = etree.fromstring(xml, parser)
# → OK. Root tag='root', child tag='xi:include' (literal, NOT namespace-expanded).
#   parser.error_log captures the namespace warning non-fatally.
```

**Important nuance the plan must handle:** under `recover=True` the literal string tag is `"xi:include"` — NOT the Clark-notation `"{http://www.w3.org/2001/XInclude}include"` that appears when `xmlns:xi` IS declared. Current `_local_name()` at `parsers/xml.py:76-80` only strips Clark notation (`tag.split("}", 1)[1]`), so a raw `xi:include` string would currently `_walk` with tag `"xi:include"` and miss the `if tag == "include":` branch. Fix must normalize both forms:

```python
def _local_name(tag: str) -> str:
    if not isinstance(tag, str):
        return ""
    if "}" in tag:                  # Clark: {ns}local
        return tag.split("}", 1)[1]
    if ":" in tag:                  # recovered: prefix:local
        return tag.split(":", 1)[1]
    return tag
```

And `<include>` attribute scan must add `href` (XInclude's actual attribute name):
```python
target = elem.get("file") or elem.get("path") or elem.get("href")
```

### Candidate fixes (tradeoffs)

| Option | How | Pros | Cons |
|--------|-----|------|------|
| **A. `recover=True` always** | Single parser config change at `xml.py:27` | Zero-risk on well-formed files (verified byte-identical output for good XML). Captures errors non-fatally. Natural path to FR-C3 partial-parse. | Lose strict-mode signal on truly malformed files — but we already catch them as `parse_error`; with recover we get *both* the error message AND the partial tree. Net win. |
| **B. Pre-inject `xmlns:xi` on strict parse** | Regex-inject before parse | Preserves Clark-notation behavior | Fragile for other undeclared prefixes (samples in real data likely include `xs:`, `xsd:`, `xs1:`, vendor prefixes). Whack-a-mole. |
| **C. Try strict, fall back to recover on namespace errors** | Two-phase parse | Keeps strict signal for "clean" files | 2× parse cost when the namespace is undeclared; added branch complexity. |

**Recommended:** Option A. `recover=True` with the `_local_name` + `href` fixes. Push the partial-parse surface (FR-C3) naturally through the same code path: we get whatever the recoverable tree contains.

### 2. "Source not shipped" code path

- **Viewer-side fetch:** `viewer/src/components/panels/RawSourceView.tsx:39-48`. URL is `source/${file.path}` (same-origin). On HTTP 404 it sets the "source not shipped — rerun crawler with --emit-sources" string.

### 3. Was default-off deliberate?

Yes — for repo hygiene. `viewer/public/source/` and `viewer/public/graph.json` are **both gitignored** (`.gitignore:5-7`). So sources never enter version control regardless of flag. Flipping the flag ON default is safe on size: dogfood measurement showed `./source/` = **1.9 MB for 63 files** on viva; `graph.json` itself only grew 38 kB (767 → 805 kB) because `mirror_sources` writes to sidecar files, not into the JSON. For a real proprietary codebase with ~1000 files and larger XML, sources could be ~30-50 MB — acceptable but worth noting in the plan. `graph.json` itself doesn't balloon because it never embeds source text.

### 4. Current viewer view/sort/grouping state

- **`filter-store.ts:6-16`** owns: `kinds: Set<FileKind>`, `hideTests: bool`, `folder: string | null`, `searchQuery: string`. No view-mode or sort fields.
- **`GraphCanvas.tsx:32-37`** reads filter-store + graph-store, runs `applyFilters` (`lib/filters/predicates.ts`) → `computeLayout` (dagre, `lib/graph/layout.ts`) → React Flow nodes/edges.
- **`FilterBar.tsx:38-82`** is where a view-mode switcher + sort controls naturally live (or its own `ViewModeBar` above it).
- **`App.tsx:83-101`** renders `GraphCanvas` unconditionally when `status === "ready"`. The viewer-mode branch point goes here.
- **Selection is already centralized:** `selection-store.ts` holds `selectedFileId` / `selectedParamKey`. Any new view component just calls `selectFile(id)` — `FileDetailPanel` renders automatically via its existing `useMemo`.

Add a `ViewMode = "graph" | "folders" | "list" | "table"` and `SortBy = "name" | "path" | "size" | "refCount" | "parseStatus"` (plus `SortDir = "asc" | "desc"`) to `filter-store.ts`. No new store needed; stays in the "what the user sees" bucket and doesn't cross the planMode boundary.

### 5. Dogfood crawl of viva (baseline)

Verified this research: `python -m crawler C:/Users/SanjayGupta/Downloads/viva --out ./dogfood-graph.json --no-timestamp` runs in **~0.5–1.3 s wall time**. Produces:

- 62 files (23 xml, 19 json, 13 yaml, 7 ini)
- 24 edges (11 unresolved)
- **3 parse errors — all intentional** (`config/broken.xml` fixtures in three locations: crawler tests, viewer e2e fixtures, viewer public mirror). **No real-world failures on viva itself today** because viva has no `xi:include` in-tree.
- `graph.json` = 767 kB (ungrouped sources) or 805 kB (with `--emit-sources`)
- `source/` sidecar = 1.9 MB / 63 files

So the dogfood test bar is: after the fix, parse-error count should stay at 3 (only the known-broken fixtures) and `netcdf.xml`-style files — to be added as a fixture — should succeed.

### 6. Perf baseline & hot loops

- **1.3 s wall time for 161 scanned / 62 parsed files on viva** → "a tad bit bad" scales linearly. For a real proprietary module of ~500 config files we'd extrapolate to ~5–10 s, which matches "a tad bit bad" from the brief. DS&A wins that realistically matter:
  - **`refs.py:57`** — `any(p.key == raw for p in src.params)` runs inside the per-ref loop. O(refs × params) per file. Fix: pre-build `{p.key for p in src.params}` once per source and use `in` → O(refs) per file.
  - **`refs.py:23-26`** — `by_param_key` build is already hash-indexed. Good.
  - **Discovery + parse is serial.** Parallelizing with `ThreadPoolExecutor(max_workers=os.cpu_count())` over `parse_file` calls should give a 2–4× speedup on wall time; lxml releases the GIL. Measure before committing.
  - **No XPath usage in `xml.py` today** (it uses element tree walking). "Cache compiled XPath" from the brief is pre-emptive — only activate if a future parser adds XPath.

## Patterns to Follow

- **Library-first, CLI thin.** New flags added to `__main__.py` only; logic stays in `crawl()`/`emit.py` (from v1-demo research).
- **Parsers independent, no cross-parser imports.** Keep XML changes inside `parsers/xml.py` (from `backend-agent.md` convention).
- **Fixture-first TDD.** Add `xi-include.xml` fixture + expected before touching parser code. Extend `sample-module.expected.graph.json` at the same time (v1-demo retrospective: fixture-first kept scope tight).
- **Zustand modular stores.** Extend `filter-store.ts`, don't create a new store unless the state crosses the plan-mode boundary (DECISIONS.md 2026-04-20).
- **View components in `components/views/`, pure sort logic in `lib/views/sort.ts`.** Respect the `components/` (UI) vs `lib/` (logic) split (CLAUDE.md Key Conventions).
- **Selection-store is the single source of truth** across all view modes. New view components call `selectFile(id)` — they don't re-implement selection (pattern from existing `GraphCanvas.tsx:31-81`).
- **Deterministic crawl output.** Any parallelization must preserve stable sort on files/edges (v1-demo TR8). Parse in parallel but collect-then-sort.
- **Offline E2E must exercise lazy paths.** Extend `offline.spec.ts` to open raw tab on a file with `--no-emit-sources` route too (v1-demo `SUMMARY.md:77`).
- **Commit prefixes:** `fix(crawler):`, `feat(crawler):`, `feat(viewer):`, `test(crawler):`, `test(viewer):`, `docs:`. **No Claude trailers** (user memory).

## Patterns to Avoid

- **Pre-injecting `xmlns:xi`** via regex — fragile, doesn't generalize to other undeclared namespaces (Option B above).
- **`multiprocessing`** for the parse pool — overkill at viva-scale, complicates Windows packaging. Use a thread pool and only if measurement shows a real win.
- **Dropping strict mode without capturing errors** — `recover=True` must still populate `parse_error` from `parser.error_log` so the existing UI surface keeps working.
- **Adding a new Zustand store for view-mode/sort** — bleeds state that plan-mode-store should not own (DECISIONS.md). Stays in `filter-store` (or a dedicated `view-store` that's still on the "live graph" side of the wall).
- **Divergent TS types on viewer when schema changes** — if partial-parse evolves `parseError` into a struct, treat it as a foundation task: update `docs/GRAPH-SCHEMA.md`, regenerate the fixture `graph.json`, update zod + TS types in lockstep.
- **View mode proliferation.** Four modes is the ceiling for this feature. Don't add a timeline/heatmap/etc.
- **Snapshot-testing React Flow / view components.** Test sort logic via unit tests on `lib/views/sort.ts`; test view switching via Playwright user flow (v1-demo pattern).
- **Embedding `emit_sources` path logic in the viewer.** Viewer just tries `source/<path>` and handles 404 gracefully.

## Recommended Approach

**One feature, four streams after a foundation task:**

1. **Foundation (blocking, small):** Pick `parse_error` shape (string, per open question), update `docs/GRAPH-SCHEMA.md` if needed, extend the `sample-module/` fixture with `xi-include.xml` + `entity-decl.xml` + update `sample-module.expected.graph.json`. Decide view-mode scope (3 vs 4) and store placement.

2. **Crawler stream [P]:** XML parser hardening (`recover=True`, `_local_name` fix, `href` capture, ENTITY decls, partial-parse), refs hash-index fix, `--emit-sources` default flip + `--no-emit-sources`, optional parallel parse with measurement. Per-change pytest coverage; dogfood integration test last.

3. **Viewer stream [P]:** View-store (or extend filter-store), view-mode switcher in chrome, FolderView + TableView components, sort lib, wire into `App.tsx` switch. "View raw anyway" in `FileDetailPanel`, reword `RawSourceView` fallback. Vitest on sort/view stores.

4. **Integration gate:** Regenerate viewer fixture `graph.json` from updated sample-module; regenerate `viewer/public/graph.json` with default-on sources against the sample-module; run full Playwright suite (new view-mode + parse-error specs + updated offline spec).

Streams 2 and 3 are genuinely parallel after foundation. Crawler changes don't block viewer work since the `graph.json` schema stays string-typed for `parseError` — the viewer iterates on UI against the existing committed fixture while the crawler improves.

## Scope: **M**

Rationale: Scoped enhancement, not greenfield — builds on solid v1 infra. Crawler changes are localized to `parsers/xml.py` + `refs.py` + one CLI flag. Viewer adds 2-3 view components + store field + sort lib + parse-error affordance — significant UI work but each piece is small. Fixtures + dogfood + E2E expansion add testing surface but reuse existing harnesses. No schema evolution planned (string `parseError` stays). Not S because of the multi-component view-mode work; not L because nothing requires cross-cutting refactor or a new foundation abstraction.

- **xi:include root cause:** `parsers/xml.py:27` uses strict `etree.fromstring(data)`. Switching to `etree.XMLParser(recover=True)` parses the failing file non-fatally (verified in research). Two supporting fixes required: (a) `_local_name` currently strips only Clark-notation `{ns}local` — must also strip `prefix:local` since recover mode keeps literal `xi:include` as the tag string; (b) include-attribute scan reads only `file`/`path` — must add `href` (XInclude's real attribute). Recommended fix = Option A (recover + normalize), no schema change.
