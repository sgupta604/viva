# Research: ghcr-publish

**Date:** 2026-04-18T15-05-00 | **Status:** research-complete

## Goal

Add a GitHub Actions workflow that auto-builds the `Dockerfile` at repo root and publishes the image to `ghcr.io/sgupta604/viva` so teammates can `docker pull` without building locally. North star: simple and low-maintenance — flaky CI breaks the whole one-line-usage UX.

## Feature Type

Cross-cutting infra (CI/CD). No crawler or viewer source touched. Thin deliverable: one workflow file plus a small `docker/README.md` edit.

## Dependencies

- **runtime-image** (complete) — the `Dockerfile` at repo root is the thing this workflow builds.
- Manual Docker smoke from runtime-image retro is still deferred. The very first CI build on `main` will implicitly cover that verification (if it fails, the orchestrator will see it before flipping the package to public).

## Requirements

### Functional
- [ ] FR1: On every push to `main`, build and push the image to `ghcr.io/sgupta604/viva` with tags `:latest`, `:main`, and `:sha-<shortsha>` (fast iteration for teammates on `:latest`).
- [ ] FR2: On pushes of git tags matching `v*.*.*`, build and push with semver tags (`:1.2.3`, `:1.2`, `:1`, `:latest`) — standard release flow.
- [ ] FR3: On `workflow_dispatch`, build and push (emergency / manual rebuild).
- [ ] FR4: On pull requests, build the image but do **not** push — catches Dockerfile regressions before merge. `push: false`, no registry login side effect.
- [ ] FR5: Tags and OCI labels are computed automatically by `docker/metadata-action@v5` from the triggering ref (no hand-rolled tag math).
- [ ] FR6: `docker/README.md` troubleshooting section is updated to remove the "build locally as primary fallback" framing once the published image is live. Build-locally stays as a last-resort fallback, not the first recommendation.

### Technical
- [ ] TR1: Workflow file at `.github/workflows/publish-image.yml` (standard GitHub convention; `.github/` does not exist yet, will be created).
- [ ] TR2: Auth via the built-in `${{ secrets.GITHUB_TOKEN }}` — no PAT, no repo secrets to provision. Requires the workflow to declare `permissions: packages: write` (and `contents: read`).
- [ ] TR3: Actions pinned to canonical, widely-used majors:
  - `actions/checkout@v4`
  - `docker/setup-buildx-action@v3`
  - `docker/login-action@v3`
  - `docker/metadata-action@v5`
  - `docker/build-push-action@v5`
- [ ] TR4: amd64 only in MVP. No `platforms:` arg in the build step, or explicitly `linux/amd64`. arm64 deferred (see Risks).
- [ ] TR5: GitHub Actions cache enabled: `cache-from: type=gha`, `cache-to: type=gha,mode=max`. Cuts rebuild from ~5 min to ~1 min.
- [ ] TR6: Image-name env (`IMAGE_NAME: sgupta604/viva`) defined once at workflow level so tags/registry/image are consistent across steps.
- [ ] TR7: Build context is repo root (`.`) with the existing `Dockerfile` and `.dockerignore`. No extra context plumbing.
- [ ] TR8: Concurrency group set so multiple pushes to `main` don't queue up redundant builds (`group: publish-${{ github.ref }}`, `cancel-in-progress: true` — except for tag pushes where we keep all).

### Constraints
- Simple. One workflow file, one job (or two small jobs at most: build-and-push + pr-build, but a single conditional job is simpler).
- Offline runtime guarantee unchanged — the workflow is a BUILD-time network consumer (npm, pip, apt). Runtime image still makes zero outbound calls. Already documented in `docker/README.md:76-94`.
- No signing (cosign), SBOM, or vuln scanning in MVP — explicit non-goals.
- No auto-semver / semantic-release. Releases = `git tag v1.2.3 && git push --tags`.

## Code Examples from Spec

No direct spec examples (this is infra, not product). Canonical workflow skeleton (from the question set, verbatim for plan-agent):

```yaml
name: Publish runtime image

on:
  push:
    branches: [main]
    tags: ['v*.*.*']
  pull_request:
  workflow_dispatch:

permissions:
  contents: read
  packages: write

env:
  REGISTRY: ghcr.io
  IMAGE_NAME: ${{ github.repository }}   # -> sgupta604/viva

concurrency:
  group: publish-${{ github.ref }}
  cancel-in-progress: ${{ github.ref != 'refs/heads/main' && !startsWith(github.ref, 'refs/tags/') }}

jobs:
  build-and-push:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: docker/setup-buildx-action@v3

      - name: Login to GHCR
        if: github.event_name != 'pull_request'
        uses: docker/login-action@v3
        with:
          registry: ${{ env.REGISTRY }}
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Compute tags + labels
        id: meta
        uses: docker/metadata-action@v5
        with:
          images: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}
          tags: |
            type=ref,event=branch
            type=ref,event=pr
            type=semver,pattern={{version}}
            type=semver,pattern={{major}}.{{minor}}
            type=semver,pattern={{major}}
            type=sha,prefix=sha-,format=short
            type=raw,value=latest,enable={{is_default_branch}}

      - name: Build & push
        uses: docker/build-push-action@v5
        with:
          context: .
          file: ./Dockerfile
          platforms: linux/amd64
          push: ${{ github.event_name != 'pull_request' }}
          tags: ${{ steps.meta.outputs.tags }}
          labels: ${{ steps.meta.outputs.labels }}
          cache-from: type=gha
          cache-to: type=gha,mode=max
```

Tag behavior at a glance:
- push to `main`: `:main`, `:latest`, `:sha-<shortsha>`
- push tag `v1.2.3`: `:1.2.3`, `:1.2`, `:1`, `:latest`, `:sha-<shortsha>`
- pull_request: built, not pushed; tags computed as `:pr-<num>` (unused on disk)
- workflow_dispatch on main: same as push-to-main

## Patterns to Follow

- **Contract-first, minimal-surface pattern.** v1-demo locked `graph.json` first (SUMMARY.md:52); runtime-image pre-resolved every non-obvious choice in research (SUMMARY.md:72-76). Same playbook here: lock the exact action versions, tag strategy, and manual-step checklist in research so implement has zero judgment calls.
- **Amd64-only MVP with documented arm64 follow-up.** Runtime-image did this (SUMMARY.md:61); mirror it exactly. Don't introduce arm64 in this feature just because buildx supports it.
- **LF line endings for shell/yaml files.** `.gitattributes` already covers `*.sh text eol=lf` (runtime-image SUMMARY.md:19). Add `*.yml text eol=lf` if not already covered — workflow files must be LF for GitHub Actions to parse reliably on Windows-authored commits.
- **Prefix convention for commits:** `feat(ci): add GHCR publish workflow`. New prefix namespace `(ci)` — consistent with existing `(viewer)`, `(crawler)`, `docs:` from CLAUDE.md:148.

## Patterns to Avoid

- **Do not bake a PAT into repo secrets.** `GITHUB_TOKEN` with `packages: write` is sufficient; adding a PAT is a long-lived credential liability.
- **Do not add Slack/email notification in MVP.** User explicitly wants simple. GitHub's default "failed workflow" email to the commit author is enough.
- **Do not write tag-math in shell.** `docker/metadata-action` already handles `:latest` + semver + sha; rolling our own is the anti-pattern that causes flakiness.
- **Do not enable multi-arch via QEMU "because it's easy".** Runtime-image research flagged lxml arm64 wheel availability as a real risk. Introducing it here would couple two independent risks and slow every CI run 2-4x.
- **Do not `tsc -b`-style composite patterns here** — N/A for YAML, but the broader lesson from v1 and runtime-image (sidecar-file shadowing, SUMMARY.md:80) is: keep build outputs ephemeral to the CI runner. Don't write any artifact back into the repo tree during CI.
- **Do not amend existing workflows** — there are none, so create cleanly. (Noted for plan-agent.)

## Open Questions

| Question | Resolution |
|----------|------------|
| Should PRs from forks also build (without push)? | **Recommend yes — default behavior is fine.** `docker/build-push-action` on fork PRs runs without the login step; build validates Dockerfile. If fork-PR build gets flaky or abused, tighten to `if: github.event.pull_request.head.repo.full_name == github.repository`. Not needed in MVP. |
| Do we want a separate `pr-validate` job that only builds, vs. a single job with conditional push? | **Single job with conditional `push:` is simpler.** Two-job split is a minor refactor if we later want different timeouts/runners per path. Keep one job for MVP. |
| Semver tag patterns — include or skip `{{major}}` (`:1`) and `{{major}}.{{minor}}` (`:1.2`)? | **Include all three (`:1.2.3`, `:1.2`, `:1`).** Matches Docker Hub / GHCR convention, zero cost. Users who want floating-major can pin `:1`. |
| Should we tag `:latest` only on tag pushes, or also on `main` pushes? | **Both.** `docker/metadata-action`'s `type=raw,value=latest,enable={{is_default_branch}}` tags `:latest` on main. Tag pushes also get `:latest` via the `type=semver` expansion (semver bumps `:latest` implicitly). Team wants fast iteration off `main`, so main-driven `:latest` is the right call. **Plan-agent should confirm user agreement: some teams prefer `:latest` = last stable tag only.** Current recommendation: `:latest` tracks `main`. |
| Pin actions to SHAs instead of `@v4`/`@v5`? | **Use major-version tags (`@v4`, `@v5`)** for simplicity. Renovate/dependabot can bump later. SHA-pinning is a security-hardening follow-up, not MVP. |
| Do we need a `timeout-minutes` on the job? | **Yes, set `timeout-minutes: 30`.** Default is 6 hours, which burns Actions minutes on a stuck build. 30 min is comfortable headroom above a cold-cache ~8 min build. |
| Dockerfile `org.opencontainers.image.version` label is hardcoded to `0.1.0` (Dockerfile:28). Should the workflow override it to match the git tag? | **Yes, optionally.** `build-push-action` supports `build-args`, and `docker/metadata-action` can emit a version label automatically. Recommend adding `labels: ${{ steps.meta.outputs.labels }}` to build step — `metadata-action` already computes `org.opencontainers.image.version` from the ref. This supersedes the hardcoded Dockerfile label at runtime (OCI labels are last-wins). Plan should verify with plan-agent whether to strip the hardcoded label from Dockerfile entirely or leave it as a build-time default. Recommend: leave Dockerfile label alone, let workflow labels override. Zero touches to `Dockerfile` is the cleaner diff. |

## Orchestrator-Owned Manual Steps (Post-Merge Checklist)

**These are NOT workflow tasks — they are human-only, one-time actions that must happen after the workflow first lands on `main`.** Flag prominently in `/finalize` summary and PR description.

1. Merge the `feat/ghcr-publish` PR to `main`.
2. Wait for the first `publish-image.yml` run on `main` to succeed (~3-5 min with cold cache).
3. Go to `https://github.com/sgupta604/viva/pkgs/container/viva` → **Package settings** (right sidebar) → scroll to **Danger Zone** → **Change visibility** → select **Public** → confirm.
4. On the same page, under **Manage Actions access**, ensure the `viva` repo is linked with role **Write** so future workflow runs can push without extra scoping. (Usually auto-created; verify.)
5. Optional: under **Package settings → Manage versions**, delete the initial `:main` and `:sha-<initial>` tags if you want a cleaner first-run history. Not required.
6. Verify end-to-end: from any machine, `docker pull ghcr.io/sgupta604/viva:latest` should succeed **without** `docker login`.

**If step 3 is skipped, every `docker pull` will 403/404 and the one-line UX is broken** even though the image exists. This is the single highest-impact gotcha.

## Recommended Approach

Single stream, single workflow file, 4-6 tasks. Ship `.github/workflows/publish-image.yml` with the canonical Docker Actions stack (checkout, setup-buildx, login, metadata, build-push), pinned to major-version tags, amd64-only, with GHA cache and PR-build validation. Keep multi-arch, signing, SBOM, and vuln-scan as explicit non-goals. The plan should also contain a Post-Merge Manual Checklist (five bullet points, lifted from above) that the orchestrator owns — the "make package public" step is the difference between "one-line usage works" and "broken out of the box."

Minor `docker/README.md` edit (one troubleshooting bullet) ships in the same PR; larger doc polish (README badge, etc.) deferred. No crawler or viewer source touched — execute-agent delegation should go to the **conductor/orchestrator directly** since CLAUDE.md:14-17 lists non-source config as an orchestrator-allowed exception. `.github/workflows/*.yml` qualifies as CI config, not application source.
