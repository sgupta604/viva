## Architecture

### Summary

Add one GitHub Actions workflow (`.github/workflows/publish-image.yml`) that builds the existing root `Dockerfile` and publishes it to `ghcr.io/sgupta604/viva` on push to `main` and on `v*.*.*` tag pushes. PR triggers build-only (no push) to self-validate the workflow and catch Dockerfile regressions. `docker/metadata-action` computes tags; `GITHUB_TOKEN` with `packages: write` covers auth â€” no PATs, no repo secrets. Flip to Public is the one manual step, owned by the orchestrator and documented in `docs/RELEASE.md`. Zero source code touched (crawler/viewer untouched); this is CI + docs only.

### Data Flow

```
[dev pushes to main / tags v*.*.*]
  -> [.github/workflows/publish-image.yml]
     -> checkout@v4
     -> setup-buildx@v3
     -> login-action@v3 (GITHUB_TOKEN, skipped on pull_request)
     -> metadata-action@v5  (auto-computes tags + OCI labels)
     -> build-push-action@v5
        - platforms: linux/amd64
        - push: ${{ github.event_name != 'pull_request' }}
        - cache-from/to: type=gha,mode=max
  -> [ghcr.io/sgupta604/viva:latest, :v1.2.3, :1.2, :1, :sha-<short>]
  -> [end-user: docker pull ghcr.io/sgupta604/viva]

PR path: build runs, image never pushed -> validates Dockerfile + workflow
main path: builds + pushes :latest, :main, :sha-<short>
tag path (v1.2.3): builds + pushes :1.2.3, :1.2, :1, :latest, :sha-<short>

First publish: ONE manual step â€” flip package visibility Private -> Public in
GitHub Packages UI (see docs/RELEASE.md). Until then, every anonymous pull 403s.
```

## Key Decisions

- **Single job with conditional `push:`** instead of separate `validate` + `publish` jobs. Simpler, one moving part; split if per-path timeout/runner needs ever diverge.
- **Major-version action pins (`@v4`, `@v5`)** not SHA-pins. Security hardening is a follow-up, not an MVP task.
- **`:latest` tracks `main`**, not "last stable tag." Research resolved this in favor of fast teammate iteration.
- **Workflow labels override Dockerfile OCI labels at build time.** Leave `Dockerfile` untouched; `metadata-action` emits `org.opencontainers.image.version` from the ref and overrides the hardcoded `0.1.0`. Cleaner diff, zero Dockerfile churn.
- **`GITHUB_TOKEN` only** â€” no PAT. Avoids long-lived credential liability.
- **Manual "make public" step stays manual** and is documented prominently. Automating GHCR visibility is possible via the REST API but adds complexity and a permissions-sensitive step for a one-time action.
- **No `.gitignore` entry for `.github/`** â€” workflow files must be committed.

## Notes for Implementer

- `.github/` does not exist yet; create both `.github/` and `.github/workflows/` as part of Task 1.1.
- Use LF line endings for the YAML file. If `.gitattributes` doesn't already cover `*.yml`, Task 1.5 adds the rule. On Windows, check `git config core.autocrlf` doesn't undo this â€” committing via git will normalize if `.gitattributes` is correct.
- `metadata-action@v5`'s tag list in Task 1.1 is verbatim from research â€” do not hand-roll tag math; let the action compute tags.
- `concurrency.cancel-in-progress` must be *false* for tag pushes (we keep all tag builds) and *true* for main/PR (latest push supersedes). The expression `${{ github.ref != 'refs/heads/main' && !startsWith(github.ref, 'refs/tags/') }}` from research is inverted â€” re-read carefully: research has `cancel-in-progress: ${{ github.ref != 'refs/heads/main' && !startsWith(github.ref, 'refs/tags/') }}` which means "cancel only when NOT main AND NOT tag," i.e., only for PRs. That's the correct semantic. Copy verbatim.
- The `type=ref,event=pr` tag in research's metadata-action skeleton is optional; since we don't push PR builds, PR tags go unused. Including it is harmless and documents intent. Keep it.
- The first post-merge workflow run IS the smoke test for runtime-image's deferred manual Docker smoke (research Dependencies section). If it succeeds, both features close out cleanly; if it fails, `/diagnose` the runtime image before making the package public.
