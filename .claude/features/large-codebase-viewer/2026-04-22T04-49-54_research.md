# Research: large-codebase-viewer

**Date:** 2026-04-22T04-49-54 | **Status:** research-complete

## Goal

Make viva usable against real 3k-file / 1.7k-XML config codebases. Two gaps: (1) the viewer currently flattens to a 1369-row column because it has no hierarchy, and (2) the crawler's explicit-pattern ref resolver finds 0 edges on a real codebase that doesn't use `<include file=...>` / `<ref id=...>` / `<import path=...>`. Land hierarchical / collapsible-cluster rendering with semantic zoom, and expand crawler ref coverage (XSD, `.d/` aggregation, logical-ID-within-explicit-declaration) so an edit-planning engineer can actually see what's connected.

## Requirements

### Functional — Crawler

- [ ] FR-C1: Extract XSD validation refs from `xsi:schemaLocation` (namespace→location pairs) and standalone `xsi:noNamespaceSchemaLocation`; emit edges with `kind="xsd"` pointing at the referenced `.xsd` file. (user-brief §in-scope-crawler; target-profile §edge-sources-2)
- [ ] FR-C2: Auto-aggregate `.d/` drop-in directories: when a file named `foo.xml` sits alongside a directory `foo.d/`, emit a synthetic edge `kind="d-aggregate"` from `foo.xml` to each child of `foo.d/`. Preserve numeric-prefix load order on child files as an edge attribute (`order: int`). Pattern detected by directory suffix, not by hardcoded name. (user-brief §in-scope-crawler; target-profile §edge-sources-4)
- [ ] FR-C3: Logical-ID matching **within explicit-declaration bounds**. Scan attributes that declare identifiers (`id=`, `model-id=`, `scheme=`, `name=` on elements with a known-declarer tag — see TR-C3 for the declarer-vs-reference distinction). Build a logical-ID index; link files whose referenced IDs match declared IDs elsewhere. Emit `kind="logical-id"`. No filename-glob inference, no cross-attribute fuzzy matching. (user-brief §in-scope-crawler, explicit line "user wants edges they can trust"; target-profile §edge-sources-3)
- [ ] FR-C4: Tolerate dangling hrefs with `xi:fallback` semantics: if an `<xi:include href="..."/>` target doesn't exist and a sibling `<xi:fallback/>` is present, record the edge as `unresolved` with `unresolved` prefix `"fallback:"` and DO NOT flag a parse error. Current behavior already records unresolved edges; this is a classification tweak. (user-brief §in-scope-crawler; target-profile §gotchas)
- [ ] FR-C5: Distinguish generated-from-template files from hand-authored when detectable. If a templating manifest (e.g. a YAML manifest listing output paths) is present AND points at a given file, set `generated: true` on the FileNode with `generatedFrom: <manifest-path>`. When not detectable, leave `generated: false` (default). Do NOT assume any specific templating system. (user-brief §in-scope-crawler; research-flagged)

### Functional — Viewer

- [ ] FR-V1: Render folders as compound / hierarchical cluster nodes (first-class, not background groups as today). A folder has a header strip, expanded/collapsed state, and a child-count badge. Clicking the header toggles expand/collapse.
- [ ] FR-V2: **Default state on load = everything collapsed to the top-level clusters.** Screen 1 shows tens of clusters, not 1400 files. (user-brief §in-scope-viewer, direct ask: "default to everything collapsed")
- [ ] FR-V3: Semantic zoom. At overview zoom (zoom < threshold Z1 ≈ 0.5), clusters render as solid tiles with aggregated edges ("N refs"). At detail zoom (zoom > threshold Z2 ≈ 1.0), children render individually. Intermediate zoom preserves cluster outline with ghost children.
- [ ] FR-V4: Cross-cluster edges remain visible at every zoom level. When both endpoints are collapsed, the edge attaches to the cluster wrapping the endpoint. Aggregate multiple source→target edges into one thicker edge with count label.
- [ ] FR-V5: A `.d/` directory cluster is **collapsed by default as a single synthetic node** (stronger default than plain folders). Expand to show the numeric-prefix-ordered children inside.
- [ ] FR-V6: Folder filter becomes a NAVIGATE action — sets focus, zooms in, auto-expands the selected cluster, preserves parent/sibling clusters around it collapsed. Stops stripping context. (user-brief §observed, screenshot 02)
- [ ] FR-V7: New edge kinds render with distinct styles: `xsd` (dashed green), `d-aggregate` (subtle gray — structural, not conceptual), `logical-id` (solid amber). Existing `include`/`ref`/`import` unchanged.
- [ ] FR-V8: "Generated" files visually distinct from hand-authored (e.g. faded border + badge). They appear in the graph but clearly signal "don't edit this directly."

### Technical

- [ ] TR-1: Stay on React Flow 11+. Use `parentNode` + `extent: 'parent'` for compound nodes (supported since RF 10, hardened in 11). No library swap. See Risks #1 for measurement plan.
- [ ] TR-2: Layout via **elkjs** (`elk.layered` with `hierarchyHandling: INCLUDE_CHILDREN`). elk natively does hierarchical layout with intra-cluster + inter-cluster edge routing. Dagre at the current flat-plus-compound config already fails at 1369 nodes (screenshot 01). Keep dagre as fallback for un-clustered views (table, explorer) if any.
- [ ] TR-3: New Zustand slice `hierarchyStore` for expand/collapse state — does NOT merge into `graph-store`, `filter-store`, `view-store`, or the future `plan-mode-store`. Keys by cluster path (POSIX). Persist to `sessionStorage` so a refresh keeps expansion state. (DECISIONS.md modular-stores rule)
- [ ] TR-4: `graph.json` schema delta (bump `version` to `2`, with viewer fallback that reads v1). New optional fields on FileNode: `generated: boolean`, `generatedFrom?: string`. New top-level `clusters: ClusterNode[]` precomputed by crawler (path, parent, childFiles[], childClusters[]) — viewer does not re-derive from file paths. New edge kinds `"xsd" | "d-aggregate" | "logical-id"` added to the `kind` union. New optional edge field `attrs?: {order?: int, ...}` for `.d/` load order.
- [ ] TR-5: Schema + new edge kinds are a **foundation stream** before crawler or viewer streams start (CLAUDE.md graph.json contract rule). docs/GRAPH-SCHEMA.md updates in the same commit.
- [ ] TR-6: Single branch `feat/large-codebase-viewer`. One PR to main. (user-brief §constraints)
- [ ] TR-7: No Claude trailers in commits. (user MEMORY.md)
- [ ] TR-8: No network, no backend, no AI, no new heavy deps except elkjs (~500 KB minified; acceptable for an offline static bundle). (CLAUDE.md)
- [ ] TR-9: **Dogfood fixture at scale:** synthesize a 3000-file / 1700-XML fixture at test time (pytest tmp_path, deterministic generator). Do NOT check in a 3k-file fixture — repo bloat. Don't depend on a public download (breaks offline constraint).
- [ ] TR-10: FPS budget. Target 30 FPS while panning with 3k nodes BUT only ~30–200 nodes visible at any one time (because default-collapsed). Measurement via Playwright + `performance.now()` in a bench spec; gate at 30 FPS on CI-class hardware.

### Constraints

- graph.json is the contract. Hierarchical cluster metadata, synthetic edges, new edge kinds are lockstep.
- Fully offline. Everything loads from disk.
- Zustand modular stores (DECISIONS.md).
- v1 principle preserved: edges must be explicit enough for users to trust them. No heuristic filename-glob match in this feature (user-brief §out-of-scope).
- Generalize, don't hardcode. No target-profile-specific paths in code.

## Code Examples from Spec

**Scale target** (target-profile-example.md): ~3000 files, ~1700 XML, plus XSD/JSON/YAML/shell. Tier tree `base-config/docstorage/{global,users,groups}/`. Validation mirror `xsd/`. Scripts dir `scripts/templating/` with `templating_config.yaml`.

**Dominant edge patterns** (target-profile-example.md §edge-sources, priority order):

1. XInclude `<xi:include href="..." xpointer="..."/>` — already parsed; xpointer resolution deferred.
2. XSD `xsi:schemaLocation`, `xmlns` → files in `xsd/`.
3. Logical IDs: `model-id`, `scheme`, patterns like `PS.557WW_SC.U_DI.C_GP.LIS-...`.
4. `.d/` drop-ins: `parameters.d/`, `resolve.d/`, `catalogue.d/`. Numeric prefix on child filenames = load order.

**Schema delta skeleton:**

```ts
// v2 additions
interface ClusterNode {
  path: string;         // POSIX, relative to root
  parent: string | null; // parent cluster path, null for top-level
  childFiles: string[];  // file ids
  childClusters: string[]; // cluster paths
  kind: "folder" | "d-aggregate"; // "d-aggregate" means the .d/ drop-in case
}
interface FileNode {
  // ...existing...
  generated?: boolean;            // default false
  generatedFrom?: string | null;  // manifest path
}
interface Edge {
  kind: "include" | "ref" | "import" | "xsd" | "d-aggregate" | "logical-id";
  attrs?: { order?: number };
}
interface Graph {
  version: 2;
  clusters: ClusterNode[];
  // ...existing...
}
```

## Patterns to Follow

- **Foundation stream first** (xml-viewer-hardening SUMMARY: "F.1 + F.2 up front … No merge conflicts"). Land graph.json schema v2 + docs/GRAPH-SCHEMA.md + types.ts + schema.ts before crawler or viewer feature work.
- **Fixture-first TDD on crawler** (xml-viewer-hardening SUMMARY §Worked-Well): write minimal XSD / `.d/` / logical-ID fixture files under `crawler/tests/fixtures/sample-module/` and failing tests before touching parsers/refs.
- **Fresh Zustand slice, no cross-store imports** (xml-viewer-hardening §worked-well on view-store.ts + DECISIONS.md §2026-04-20). `hierarchyStore` must not import from `filter-store`/`graph-store`/`view-store`; consumers compose at component level.
- **Fixed-dimension invariants exported from single source** (post-finalize lesson, FileNode width fix 26f948f). Cluster dimensions and collapsed-cluster tile size exported as named constants from `layout.ts`, imported by `ClusterNode.tsx` — no CSS-vs-TS duplication.
- **Dogfood integration test asserts structural invariants, not exact counts** (xml-viewer-hardening SUMMARY). At scale, assert upper bounds ("<=2s layout", ">=1 XSD edge produced on fixture"), never exact edge counts.
- **Re-entrant test coverage** (xml-viewer-hardening post-finalize, dd2f273 emit-sources self-feedback). If `.d/` aggregation or cluster building emits into the crawl tree somehow, run crawler twice to catch feedback loops. (Unlikely here, but the lesson stands — add a re-crawl invariant to the integration test.)
- **Error-on-schema-key-mismatch** (post-finalize lesson: parse_error vs parseError). ClusterNode field naming must be explicit: `childFiles` not `child_files` in JSON (camelCase in docs/GRAPH-SCHEMA.md, snake_case in Python dataclass with `.to_dict()` shim).
- **Crawler CLI preserves default=off for new expensive features if any.** Templating detection (FR-C5) is optional and narrowly-scoped — enable always-on because it's cheap (one YAML file scan); XSD resolution is also always-on. `.d/` aggregation is always-on. No new CLI flags needed unless a measurement shows otherwise.

## Patterns to Avoid

- Do NOT hardcode `.d/` directory names (`parameters.d`, `resolve.d`, `catalogue.d`). Detect by suffix `.d` on the directory name AND presence of a sibling file (`foo.d` paired with `foo.xml`). Generalizes cleanly.
- Do NOT hardcode `docstorage/global/` or `xsd/` paths. XSD resolution is driven by `xsi:schemaLocation` content in the XML, not directory position. Logical-ID indexing runs over every file, not just files in specific folders.
- Do NOT infer filename-glob logical-IDs (target-profile called this out; user-brief §out-of-scope explicitly forbids). Only link when file A declares `id="X"` on an attribute AND file B references `id="X"` in an attribute with a known-reference name. The user's rule: "edges I can trust."
- Do NOT try to support XPointer fragment resolution (out-of-scope, user-brief §out-of-scope).
- Do NOT swap React Flow for Cytoscape / Sigma / d3 mid-feature. The FolderGroup.tsx infrastructure already anticipated compound nodes; fulfilling that anticipation is cheaper than a library swap. See Risks #1.
- Do NOT keep the current folder-filter-as-HIDE behavior — it's the bug (screenshot 02).
- Do NOT embed Jinja2 or jinja2-parser logic. Template introspection is out-of-scope; we only note "this file was listed as a generated output in a manifest," nothing about what it templated to.
- Do NOT re-layout the graph on every pan/zoom. Layout is computed once per graph-change or per expand/collapse; semantic zoom is a pure CSS/visibility toggle on top of laid-out positions.

## Risks

| Risk | Severity | Mitigation |
|------|----------|------------|
| 1. React Flow can't hit 30 FPS at 3k nodes even with collapsed clusters. RF renders every node in DOM; at 3k files it's a known stressor (>1500 DOM nodes is where people swap to Cytoscape canvas). | H | Default-collapsed means <200 visible at a time in typical interaction. Add a virtualization guard: hide cluster children from `nodes[]` array when cluster is collapsed (don't just `hidden: true`). Add a Playwright FPS bench early in implementation (foundation phase) on a 3k synthetic fixture — if <30 FPS, fall back to Cytoscape before building the full UX. |
| 2. elkjs layout is slow at 3k+ nodes. Known to be 1–5 s on large graphs. | M | Run elk incrementally: full layout only on graph-load; on expand/collapse, re-layout only the affected subtree using elk's `interactiveLayout` option. Web Worker the elk call (elkjs ships an `ELK.Worker` variant). |
| 3. Logical-ID indexing produces huge amount of noise on a real codebase — every `id="1"` maps to hundreds of files and renders as spaghetti. | M | Require a minimum specificity: skip IDs that are pure integers, single letters, or appear in more than N files (N=20 tuneable; maybe config flag `--logical-id-max-cardinality=20`). Surface skipped-IDs as diagnostics rather than silent drops. Bound total logical-id edges to e.g. 10 × file_count to prevent schema explosion. |
| 4. XSD resolution finds wrong target when two schemas share a tail name. | L | Follow precedence from refs.py: path-relative first, then absolute, then global match. Accept ambiguous → unresolved with `"ambiguous:"` prefix as existing code does. |
| 5. `.d/` detection false-positives (e.g. `node_modules/mod.d/` is NOT a drop-in dir in viva's sense — but we already prune `node_modules`; broader risk is any `*.d/` dir on the path that happens to have a `.xml` sibling). | L | Require **both** conditions: dir ends in `.d/` AND sibling file with matching stem exists. If only one side exists, don't aggregate. Document rule in docs/GRAPH-SCHEMA.md. |
| 6. Generated-template detection produces false positives — template manifest is project-specific. | M | Make it strictly opt-in via **detection heuristic + evidence only**: only mark generated when the manifest EXPLICITLY lists the file by relative path. Do NOT apply glob-based inference. If a user has no template manifest, no files are flagged — zero harm. |
| 7. Schema v2 migration breaks existing users who have old graph.json on disk. | L | Viewer schema.ts validates both v1 and v2; v1 graphs parse with clusters[] derived from file paths and empty/default new fields. This is already on the list for schema.ts changes (TR-4). |
| 8. Semantic-zoom edge aggregation correctness — aggregating N edges between two clusters into "N refs" can hide mixed-kind information (xsd+logical-id mixed in same bundle). | M | Aggregated edge shows kind breakdown on hover tooltip. At overview zoom, use dominant kind's color with count. At detail zoom, split into per-kind edges. Document in EdgeStyles.ts. |
| 9. Folder-filter-as-NAVIGATE breaks existing Playwright specs that assert hide behavior. | L | 10 filter-related E2E assertions to rewrite. Check `viewer/e2e/filters.spec.ts` and update in-lockstep with the semantics change. Treat as part of V-stream tasks, not a separate step. |
| 10. elkjs + React Flow integration has no prior art in this codebase. Import/bundle-size may surprise. | M | Prototype in a spike task early in implementation. elkjs ships as an ES module (~300 KB gzipped). If it doesn't integrate cleanly, fall back to: dagre-at-cluster-level + separate rendering of inter-cluster edges, which is strictly-speaking enough for FR-V1..V6 without a hierarchical solver. |

## Recommended Approach

Land the change as a single-branch foundation-then-parallel-stream effort:

**(a)** Foundation stream (F) locks graph.json v2: docs/GRAPH-SCHEMA.md + graph.py dataclasses + viewer types.ts + schema.ts + fixture regen for v2 + a FPS-bench baseline harness. This unblocks crawler and viewer work independently.

**(b)** Crawler stream (C) in parallel after F: C.1 XSD refs, C.2 `.d/` aggregation + ClusterNode build, C.3 logical-ID indexing with specificity cap, C.4 xi:fallback classification, C.5 templating manifest detection. Each is fixture-first TDD with a minimal failing test before implementation.

**(c)** Viewer stream (V) in parallel after F: V.1 hierarchyStore Zustand slice, V.2 elkjs layout integration (behind feature flag first), V.3 ClusterNode component with expand/collapse, V.4 semantic zoom rendering toggle, V.5 edge aggregation, V.6 folder-filter → navigation refactor, V.7 new edge-kind styles, V.8 generated-file styling.

**(d)** Integration stream (I): synthesize 3k-file fixture, wire Playwright large-scale.spec.ts + FPS bench as hard gate, dogfood against the user's real codebase one more time, update filters.spec.ts for new NAVIGATE semantics.

Ship behind no feature flag — this IS the new default. Old v1 graph.json still loads via the schema.ts fallback so users with stale fixtures aren't blocked.

## Scope: **XL**

Two substantive subsystems land in lockstep (viewer compound-hierarchy renderer + crawler three-new-edge-kinds + ClusterNode schema), plus a new elkjs layout path, plus FPS-bench infrastructure, plus semantics change on the folder filter. Comparable in shape to xml-viewer-hardening (Scope L) but larger because (1) new layout library, (2) new rendering primitive (cluster nodes are a step change from the current flat + faded-background approach), (3) schema version bump, (4) scale test harness creation. Expect 30–45 plan tasks across F/C/V/I streams. Recommend carving into **two rounds** if user wants a smaller first PR, where round-1 ships Foundation + Viewer hierarchy at existing crawler behavior (user can at least navigate their 1432 files) and round-2 ships the new edge kinds. **Default recommendation:** single XL PR — the value depends on doing both halves.
