## Goal
Deliver the full v1 demo of the viva config-codebase visualizer: a Python crawler that parses XML/YAML/JSON/INI and emits `graph.json`, plus a static React viewer that renders a grouped, interactive map with file detail panel, parameter highlighting, search, filters, and dark mode. Target a focused fixture codebase of ~20–50 files. Offline-only.

## Feature Classification
**Cross-cutting, greenfield.** Spans crawler + viewer + shared `graph.json` contract + fixtures. First feature in the project — no retrospectives to review (`.claude/features/` empty aside from this directory). Contract-first: `graph.json` schema is the foundation stream both sides depend on.

## Code Examples from Spec / Agent Docs

### Proposed `graph.json` schema (to lock as foundation task)
Based on `backend-agent.md:150-170` with minor additions to support viewer filters/param-highlighting:
```json
{
  "version": 1,
  "root": "fixtures/sample-module",
  "generatedAt": "2026-04-18T14:30:00Z",
  "files": [
    {
      "id": "abc123",
      "path": "config/radar.xml",
      "name": "radar.xml",
      "folder": "config",
      "kind": "xml",
      "sizeBytes": 2048,
      "params": [
        { "key": "radar.threshold_rain", "value": "0.25", "line": 42, "kind": "scalar" }
      ],
      "parseError": null
    }
  ],
  "edges": [
    { "source": "abc123", "target": "def456", "kind": "include" },
    { "source": "abc123", "target": null, "kind": "ref", "unresolved": "ghost-file.xml" }
  ]
}
```
Notes:
- `id` = stable hash of `path` (e.g., first 10 hex chars of sha1).
- `path` is POSIX-style relative to `root` even on Windows (TR9).
- `name`/`folder` are derivable from `path` but shipped to keep viewer code simple and search index cheap.
- `params[].kind` is `"scalar" | "list" | "map"`; complex values stringified with depth limit.
- `edges[].kind` is `"include" | "ref" | "import"`.
- `generatedAt` is the only non-deterministic field — isolate it so byte-determinism holds when excluded.

## Patterns to Follow
- **`crawl()` library-first, CLI thin wrapper** (`backend-agent.md:21-56`) — tests import `crawl()` directly.
- **Parsers are independent and swappable** — no parser imports another; lxml types don't leak past `parsers/` (`backend-agent.md:14-15`).
- **Viewer logic/UI split** — `src/lib/` owns logic, `src/components/` is presentation only (`frontend-agent.md:13-14`).
- **Zustand for cross-component state, `useState` for local** (`frontend-agent.md:33-36`).
- **Validate `graph.json` shape at the boundary** in the viewer (zod or hand-rolled guard) with actionable error UI (`frontend-agent.md:88,123`).
- **Fixture-first TDD** — write fixture file + expected parse result before parser code (`backend-agent.md:202`).
- **Deterministic output** — stable sort, stable IDs, so diffs on `graph.json` are meaningful.
- **Commit prefixes:** `feat(crawler):`, `feat(viewer):`, `test(crawler):`, `docs:` (CLAUDE.md).

## Patterns to Avoid
- **Implicit reference matching** — out of scope for v1, will cause false positives and tuning work (spec non-goals).
- **Barrel files / CSS modules / styled-components** (`frontend-agent.md:17-18`).
- **Storing the ReactFlow instance in component state** — use `useReactFlow()` hook (`frontend-agent.md:60`).
- **Snapshot-testing components or asserting on React Flow internals** — brittle; test via store/logic (`frontend-agent.md:98`).
- **Embedding Monaco eagerly** — it's heavy; lazy-load (`frontend-agent.md:66`).
- **`print()` in crawler library code** — use `logging` (`backend-agent.md:18`).
- **Aborting the crawl on a malformed file** — capture on `parse_error`, keep going (`backend-agent.md:81`).
- **Divergent local types on the viewer when schema changes** — stop and raise a foundation task (`frontend-agent.md:129-131`).
- **Using OS-specific `Path` separators directly in `graph.json`** — always POSIX-normalize for cross-OS consistency.

## Recommended Approach

Execute as a **single feature with three streams after a foundation task**:

1. **Foundation (blocking):** Lock `graph.json` schema -> `docs/GRAPH-SCHEMA.md` + shared types. Seed `crawler/tests/fixtures/sample-module/` (build the fictional radar module). Set up root `.gitignore` and toolchain skeletons (crawler `pyproject.toml`, viewer `package.json` + Vite + Tailwind + shadcn init).
2. **Crawler stream [P]** (backend-agent): Graph model -> parsers (XML/YAML/JSON/INI, one at a time, TDD from fixtures) -> `resolve_references` -> `emit.to_json` -> CLI. Per-parser unit tests; one integration test that walks `sample-module/` and snapshots the resulting `graph.json` against a committed expected file.
3. **Viewer stream [P]** (frontend-agent): `loadGraph` + zod validator against a committed `e2e/fixtures/graph.json` (identical to crawler expected output) -> state stores -> GraphCanvas with React Flow + dagre grouped by folder -> FileDetailPanel (ParamTree + lazy Monaco) -> SearchPalette -> FilterBar -> param-highlight logic -> dark-mode chrome.
4. **Integration gate:** Regenerate `graph.json` from the sample-module fixture via crawler, copy into `viewer/public/`, run full Playwright suite that exercises every FR-V.

This ordering matches WORKFLOW.md's contract-first stream pattern and lets the two streams run truly in parallel once the schema and fixture `graph.json` exist. Parallelism is the only realistic way to fit v1 into a reasonable scope.
